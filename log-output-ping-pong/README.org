** Notes & Key Takeaways for GKE

*** Container-Native Load Balancing (NEGs)
Google Cloud Engine (GKE) requires an explicit opt-in for Container-Native Load
Balancing on custom NodePort services to route traffic directly to Pod IPs via
Network Endpoint Groups (NEGs). Without this, GKE uses legacy NodePort hops,
which can cause connection resets (~curl: (52) Empty reply from server~).

Add this annotation to ~service.yaml~:

#+begin_src yaml
metadata:
  annotations:
    # Enables Container-Native Load Balancing on GKE
    cloud.google.com/neg: '{"ingress": true}'
#+end_src

*** Storage Class Compatibility
- Local setups (~k3d~ / ~k3s~) default to ~local-path~ as the ~storageClassName~.
- GKE defaults to ~standard-rwo~ (or ~standard~).
- *Tip:* Omitting ~storageClassName~ or setting it to ~""~ in ~volumeClaimTemplates~
  allows Kubernetes to automatically fallback to the default StorageClass of
  whatever cluster it's deployed to.

*** GKE Ingress Path Matching
GKE's default ~gce~ Ingress requires explicit wildcard definitions for routes
(~/log~ and ~/log/*~) using ~pathType: ImplementationSpecific~ to ensure subpaths and
trailing slashes route smoothly without dropping traffic.
