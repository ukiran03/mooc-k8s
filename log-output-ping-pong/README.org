*** Key Notes for Future Self

- Gateway API Rule Structure :: Always ensure each endpoint target in an
  =HTTPRoute= is defined as a separate item in the =rules= array. Grouping multiple
  =matches= or =backendRefs= under a single rule block causes silent routing
  conflicts.

- Path Forwarding vs. Rewriting :: By default, GKE Gateway forwards the exact
  request path (e.g., =/log= or =/pingpong=) to the container. If the application
  server expects traffic at the root path (=/=), you must explicitly configure a
  =URLRewrite= filter in the =HTTPRoute= rule.

- GKE Load Balancer Health Checks :: GKE provisions separate GCP Backend
  Services per target port. Ensure both apps expose an unauthenticated health
  endpoint (e.g., =/health=) and attach a =HealthCheckPolicy= targeting the service
  using =USE_SERVING_PORT=.

- Debugging "Empty reply from server" (curl error 52) :: If curl returns an
  empty response immediately, check:
  1. Are GCP health checks failing? (=kubectl describe httproute <name>=)
  2. Is the container server listening on =0.0.0.0= instead of =127.0.0.1=?
